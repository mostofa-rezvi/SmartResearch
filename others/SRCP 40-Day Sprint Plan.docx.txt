This document is the complete, day-by-day development plan for the Smart Research Collaboration Platform (SRCP). Every day includes a user story, concrete technical tasks with file names and code snippets, and a clear deliverable. The plan is designed for a single developer working full-time.

Day
11
	CF+CB Hybrid Matching Engine
Phase 2
	SRCP 40-Day Plan
Mostofa Aminur Rashid
	

User Story:  As a new researcher, I want the platform to suggest relevant collaborators on first login based on my profile tags.
	

Technical Tasks
#
	Area
	Task / Command / Code Snippet
	1
	CBF (Content-Based)
	routers/match.py: get user embedding; ES kNN search for top-20 similar researchers. Weight: cosine(embedding_i, embedding_j) → content_score.
	2
	CF (Collaborative)
	Build user-item interaction matrix from project_members + paper co-authors. Compute item-item cosine similarity. If no interactions yet: CF weight = 0.
	3
	Hybrid blend
	final_score = alpha * content_score + (1-alpha) * cf_score. Alpha = 0.8 (cold start) → 0.5 (experienced). Store alpha_lookup in user record.
	4
	POST /match
	Accept { user_id }. Return list[{ user_id, score, reason: 'shared interests: NLP, CV' }]. Top 10 results.
	5
	Schedule on signup
	Node API: after register, fire async POST to ML /match; store results in Redis as rec:{user_id} TTL 3600.
	6
	Refresh job
	Node cron (node-cron): every 6 hours, for active users call /match and refresh Redis cache.
	

Deliverable:  CF+CB matching returning top-10 collaborator suggestions; cold-start works with only profile tags; results cached in Redis.
	

Trade-off / Note:  MVP: In-memory matrix, 1000 user limit. Post-MVP: Spark/distributed CF for 100k+ users.

13
	TrustRank Scoring Algorithm
Phase 2
	SRCP 40-Day Plan
Mostofa Aminur Rashid
	

User Story:  As a researcher, I want scores that reflect real credibility so that spam profiles appear at the bottom.
	

Technical Tasks
#
	Area
	Task / Command / Code Snippet
	1
	Graph model
	routers/trustrank.py: represent researchers as nodes; edges = citation (weight 2) + co-author (weight 1.5) + endorsement (weight 1) + institutional_verify (weight 3).
	2
	PageRank impl
	Use networkx.pagerank(G, alpha=0.85, personalization=seed_nodes). Seed nodes = verified professors + DOI-verified papers.
	3
	Score normalize
	Normalize scores to [0,1]; store in users.trust_score via UPDATE query. Log: 'Trust scores updated for N users'.
	4
	POST /trust/compute
	Admin-only endpoint. Triggers full recompute. Returns { updated: N, duration_ms: X }.
	5
	Display
	Add trust badge to RecommendationCard: green shield for >0.7, yellow for 0.4-0.7, none below. Tooltip explains score.
	6
	Verified badge
	Backend: POST /users/:id/verify (admin only). Sets verified=true. Frontend: blue checkmark on profile.
	

Deliverable:  TrustRank computes scores for all seeded users; badges display correctly on frontend; verified admin endpoint working.
	

Trade-off / Note:  MVP: NetworkX in-process (fine for <5000 nodes). Post-MVP: Neo4j graph traversal for large-scale.

MVP vs. Post-MVP Trade-off Decisions
These architectural decisions were consciously made to deliver a working product in 40 days. Each can be upgraded post-MVP with the upgrade path documented below.


Feature
	MVP Choice
	Post-MVP Upgrade
	Rationale
	Graph DB
	NetworkX (in-process)
	Neo4j (proper graph DB)
	NetworkX fine for <5K users; add Neo4j when graph traversal becomes bottleneck
	Message Queue
	Direct HTTP calls (sync)
	Apache Kafka (async)
	Kafka adds ops complexity; direct calls fine for <100 req/s
	Infrastructure
	Docker Compose on 1 VPS
	Kubernetes (K8s)
	K8s overhead unjustified for MVP; add when you have 3+ replicas per service
	Semantic Models
	all-MiniLM-L6-v2 (90MB)
	allenai-specter (fine-tuned academic)
	MiniLM is 5x faster; upgrade if semantic relevance needs improvement
	CF Algorithm
	In-memory cosine similarity
	Spark ALS distributed CF
	In-memory fine for <1000 users; Spark needed at 100K+
	Email Service
	Gmail SMTP
	AWS SES / SendGrid
	Gmail has 500/day limit; upgrade before launch to researchers
	File Storage
	Local disk (VPS)
	S3 / R2 object storage
	Local disk = no horizontal scaling; move to S3 if adding more servers
	Search
	Elasticsearch single-node
	ES cluster (3 nodes)
	Single node = no redundancy; cluster when availability SLA matters
	Database
	PostgreSQL (single)
	PG + read replica
	Read replica needed at >10K concurrent users for analytics queries
	Real-time Docs
	Y.js in-memory + DB snapshot
	Liveblocks / PartyKit managed CRDT
	Self-hosted Y.js works; managed service removes ops burden post-MVP
