## 1. System Classification

**System Type:** Extraction / Embedding Microservice

**Description:**
A Python FastAPI microservice that generates dense vector embeddings from text. It exposes a synchronous/asynchronous `/embed` endpoint and a Redis Streams consumer. It takes researcher profile text or paper abstracts, passes them through a pre-trained Transformer model, and outputs a 768-dimensional vector to be stored in Elasticsearch and cached in Redis. "Good" looks like consistently fast response times (< 100ms) without blocking the API, and accurate semantic representation of research topics.

**Critical Failure Modes:**
1. **Event Loop Blocking:** Running CPU-bound ML inference synchronously in the main FastAPI thread, causing the entire service to hang.
2. **Memory Leaks/OOM:** Loading multiple instances of the model into RAM or improperly batching Redis Stream messages, crashing the container.
3. **Data Loss in Sync:** Consuming a Redis Stream message but failing to update Elasticsearch without a retry mechanism.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Academic Research & Collaboration Tools

**User Population:** Researchers, academics, and R&D professionals seeking collaborators and relevant literature.

**Stakes Level:** Medium (matches are suggestions, but consistently poor matches destroy platform trust).

**Output Consequence:** The generated embeddings dictate the quality of Hybrid Search and Graph pathfinding. Bad embeddings mean researchers won't find relevant peers.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|-----------------------|--------------------|--------|--------|

## 2. Framework Decision

**Selected Framework:** `sentence-transformers`

**Version:** `~2.6.0`

**Rationale:**
For a Phase 6 MVP, `sentence-transformers` provides the best balance of ease-of-use and quality. It natively handles tokenization, mean-pooling, and normalization out-of-the-box. We do not need the generic capabilities of Hugging Face `pipeline`, and we do not yet need the extreme production latency optimizations of Hugging Face `Optimum` (ONNX compilation) until we hit scale bottlenecks.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Hugging Face `pipeline` | Too generic; requires manual pooling and normalization logic to get good sentence embeddings. |
| Hugging Face `Optimum` | Premature optimization. Adds significant compilation/build complexity (ONNX/TensorRT) that isn't required for the initial integration. |

**Vendor Lock-In Accepted:** No. The model runs locally via open-source weights from the Hugging Face Hub.

## 3. Framework Quick Reference

> Fetched from official docs by `gsd-ai-researcher`. Distilled for this specific use case.

### Installation
```bash
pip install sentence-transformers fastapi uvicorn redis pydantic
```

### Core Imports
```python
from sentence_transformers import SentenceTransformer
from fastapi import FastAPI
from pydantic import BaseModel
import asyncio
```

### Entry Point Pattern
```python
app = FastAPI()
# Load model globally on startup to prevent cold-starts
model = SentenceTransformer('all-mpnet-base-v2') 

class EmbedRequest(BaseModel):
    text: str

@app.post("/embed")
async def embed(req: EmbedRequest):
    # CRITICAL: Offload CPU-bound task to threadpool
    vector = await asyncio.to_thread(model.encode, req.text)
    return {"vector": vector.tolist()}
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `SentenceTransformer` | Wrapper class for the transformer model. | Instantiated once at app startup. |
| `model.encode()` | Method to convert string/list of strings to embeddings. | Inside the endpoint or Redis stream consumer. |
| `asyncio.to_thread()` | FastAPI wrapper to run blocking code. | Whenever calling `model.encode()` in an `async def` route. |

### Common Pitfalls
1. **Sync in Async:** Calling `model.encode()` directly inside an `async def` route will block the server.
2. **Cold Starts:** Instantiating `SentenceTransformer` inside the route handler instead of globally on app startup.
3. **No Truncation:** Passing text longer than the model's `max_seq_length` (usually 384 or 512) without explicit truncation or chunking.

### Recommended Project Structure
```
src/
├── main.py           # FastAPI app and routes
├── ml_model.py       # SentenceTransformer singleton logic
├── stream_worker.py  # Redis Streams consumer loop
└── config.py         # Env vars (Redis URI, Model Name)
```

## 4. Implementation Guidance

**Model Configuration:**
- **Model:** `all-mpnet-base-v2` (generates 768-dimensional vectors).
- **Device:** `cpu` (unless a GPU is explicitly attached to the Docker container).

**Core Pattern:**
- Dual-interface: 
  1. A REST `/embed` endpoint for real-time frontend/backend requests.
  2. A background worker task polling Redis Streams for `profile.created` events to batch-embed existing records.

**State Management:**
- Redis will act as both the message broker (Streams) and a caching layer (to avoid re-embedding identical text).

**Context Window Strategy:**
- For profiles, take the first N tokens (truncate). For papers, we will need a chunking strategy in a future phase, but for now, extract and embed only the Abstract.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs.

### Structured Outputs with Pydantic

Ensure the API boundary is strictly typed.

```python
from typing import List
from pydantic import BaseModel, Field

class EmbedRequest(BaseModel):
    text: str = Field(..., min_length=10, description="Text to embed")

class EmbedResponse(BaseModel):
    vector: List[float] = Field(..., description="768-dim dense vector")
    model: str = Field(default="all-mpnet-base-v2")
```

### Async-First Design

Because `sentence-transformers` uses PyTorch under the hood, its operations are strictly blocking (CPU bound). 
**Rule:** You must wrap `encode` calls in a threadpool to allow FastAPI to handle other HTTP requests or health checks simultaneously.

### Prompt Engineering Discipline

*Not applicable for embedding models (no prompts involved).*

### Cost and Latency Budget

- **Cost:** Free (self-hosted).
- **Latency Target:** < 150ms per single text string on CPU.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| Inference Latency | < 150ms (Pass), > 150ms (Fail) | Code (Locust / API benchmark) | Critical |
| Semantic Quality (NDCG) | Top 5 matches are highly relevant | MTEB Benchmark / Human review | High |
| Worker Throughput | Batch processing > 50 profiles/sec | Code | Medium |

### Eval Tooling

**Primary Tool:** MTEB (Massive Text Embedding Benchmark) for offline model validation, plus standard API load testing tools (e.g., `locust` or `wrk`) for latency.

**Setup:**
```bash
pip install mteb
```

### Reference Dataset

**Size:** 50 real-world researcher profiles and paper abstracts.

**Composition:**
- 20 profiles from Computer Science.
- 20 profiles from Biology/Chemistry (to test specialized terminology).
- 10 edge cases (empty profiles, non-English text).

**Labeling:**
- Synthetically generated ground truth via LLM, verified by human review for the top 5 nearest neighbors.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Empty Text | `len(text) == 0` | Block (Return 400 Bad Request) |
| Max Tokens Exceeded | String length > 2000 chars | Truncate text before embedding to prevent memory spikes |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| Stream Consumer Lag | Monitor Redis `XPENDING` | Auto-scale worker containers or increase batch size |
| Average Latency | Prometheus scraping | Optimize with Hugging Face `Optimum` (ONNX) if latency degrades |

## 7. Production Monitoring

**Tracing Tool:** Prometheus/Grafana (Metrics) + standard JSON logging.

**Key Metrics to Track:**
- `embed_latency_seconds` (Histogram)
- `redis_stream_lag` (Gauge)
- `batch_size` (Histogram)

**Alert Thresholds:**
- P95 Latency > 300ms.
- Redis Stream consumer lag > 1000 messages.

**Smart Sampling Strategy:**
- Log 100% of errors/exceptions.
- Log 5% of embedded text inputs for offline quality review (ensure PII is scrubbed).
