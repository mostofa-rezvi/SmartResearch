# Phase 2 Security Audit: Auth System & Token Management

**Audited:** 2026-04-25
**Scope:** Token storage, CSRF, XSS, OAuth callback validation, password hashing, rate limiting
**Remediation:** Applied inline via `--chain`

---

## Threat Matrix

| # | Threat | Severity | Status | Fix Applied |
|---|--------|----------|--------|-------------|
| T1 | **OAuth callback open redirect** | 🔴 CRITICAL | ✅ FIXED | Whitelisted `ALLOWED_REDIRECT_ORIGINS`; all redirects validated against it. |
| T2 | **Access token leaked in URL fragment** | 🟠 HIGH | ✅ FIXED | Replaced `#token=` with short-lived auth code (60s, single-use) stored in Redis. Frontend exchanges via `POST /exchange-code`. |
| T3 | **CSRF on cookie endpoints** | 🟠 HIGH | ✅ MITIGATED | `sameSite: lax` + all state-changing endpoints are POST. Documented as acceptable. |
| T4 | **OAuth state parameter missing** | 🟠 HIGH | ✅ FIXED | Both Google and GitHub flows now generate a `crypto.randomBytes(32)` state, stored in Redis (5m TTL), validated on callback. |
| T5 | **No password complexity validation** | 🟡 MEDIUM | ✅ FIXED | Enforced: min 8 chars, ≥1 uppercase, ≥1 lowercase, ≥1 digit. |
| T6 | **Verification token has no expiry** | 🟡 MEDIUM | ✅ FIXED | Token stored in Redis with 24h TTL. Verified against Redis before DB update. |
| T7 | **Invite accept uses 10-round bcrypt** | 🟡 MEDIUM | ✅ FIXED | Replaced inline `bcrypt.genSalt(10)` with centralized `hashPassword()` (12 rounds). |
| T8 | **`console.error` in production** | 🟢 LOW | ✅ FIXED | Replaced with `logger.error()` structured logging. |
| T9 | **OAuth account linking without consent** | 🟡 MEDIUM | ✅ FIXED | If email matches a `local` account, OAuth login is rejected with a message to link from settings. |
| T10 | **No `Secure` flag on cookie in dev** | 🟢 LOW | ✅ BY DESIGN | Documented. `secure: true` only in production. |
