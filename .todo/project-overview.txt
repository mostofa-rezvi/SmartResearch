────────────────────────────────────────────────────────────────────────────
  PROJECT CONTEXT (from SRCP Project Overview)
────────────────────────────────────────────────────────────────────────────

  End-to-End System Flow:
  ① User Registration → ② Core Engine Processing → ③ AI Matching →
  ④ Feature Modules → ⑤ Publication Assistant → ⑥ Output Delivery

  INPUT:   Profile · Papers · Interests · Skills · Behaviour signals
  PROCESS: Matching · Semantic Search · Graph Traversal · Async Queue
  OUTPUT:  Research Team · Structured Project · Published Paper

  Architecture Layers:
  - Presentation: Next.js (SSR) + Tailwind CSS + Socket.IO + Cloudflare
  - Business Logic: Node.js/Express API + Python FastAPI ML Service
  - Data: PostgreSQL → Neo4j → Elasticsearch → Redis

  Competitive Advantage (vs ResearchGate, Academia.edu, Overleaf, Semantic Scholar):
  ✔ Smart CF+CB+BERT collaborator matching (none have this)
  ✔ Full collaboration workspace with tasks, docs, milestones
  ✔ Publication assistant with journal finder + templates
  ✔ TrustRank-powered community forum
  ✔ Cold-start resilient: CBF bootstraps from tags on day one

────────────────────────────────────────────────────────────────────────────
  KEY CHANGE: Using Redis (Pub/Sub + Streams) instead of Apache Kafka
────────────────────────────────────────────────────────────────────────────
  - Redis Pub/Sub replaces Kafka for event-driven messaging
  - Redis Streams replaces Kafka topics for durable event logs
  - Simplifies infrastructure (already have Redis for caching)
  - Reduces Docker footprint and operational complexity

================================================================================
  SECTION THREE: RECOMMENDATIONS & ARCHITECTURAL CHANGES
================================================================================

1. KAFKA → REDIS STREAMS (CONFIRMED)
   - Replace all "Kafka" references with Redis Streams (XADD/XREADGROUP)
   - Use ioredis (already installed) for Streams + Pub/Sub
   - Consumer groups via XREADGROUP for durable processing
   - Benefits: One fewer Docker service, simpler ops, sufficient
     throughput for research platform scale

2. ADD BullMQ FOR JOB QUEUES (RECOMMENDED)
   - Use BullMQ (Redis-backed) for background jobs:
     batch embeddings, email sends, PDF generation
   - Separates job queue concerns from event streaming
   - npm install bullmq

3. ADD PRISMA ORM (RECOMMENDED)
   - Replace raw pg queries with Prisma for type safety
   - Auto-generates TypeScript types from PostgreSQL schema
   - Migration management built-in
   - npm install prisma @prisma/client

4. ADD ZUSTAND FOR FRONTEND STATE (RECOMMENDED)
   - Lightweight state management for Next.js
   - Better than Context API for complex state (auth, search, workspace)
   - npm install zustand

5. ADD ZDOG/RECHARTS FOR DATA VISUALIZATION (RECOMMENDED)
   - TrustRank scores, collaboration graphs, search analytics
   - npm install recharts

6. ADD DOCKER HEALTH CHECKS (RECOMMENDED)
   - Add healthcheck directives to each Docker service
   - Proper depends_on with condition: service_healthy

7. ADD OPENTELEMETRY TRACING (RECOMMENDED)
   - Distributed tracing across Node.js ↔ FastAPI ↔ databases
   - Critical for debugging cross-service issues
   - npm install @opentelemetry/sdk-node

8. CONVERT BACKEND TO TYPESCRIPT (OPTIONAL)
   - Backend currently uses CommonJS JavaScript
   - TypeScript would catch bugs earlier and improve DX
   - Can be done incrementally with allowJs

9. ADD RATE LIMITING TO ML SERVICE (RECOMMENDED)
   - FastAPI embedding endpoint needs rate limiting
   - Prevent abuse of expensive GPU/CPU inference
   - Use slowapi or custom Redis-based limiter

10. ADD REDIS SENTINEL FOR HA (PRODUCTION)
    - Since Redis now handles caching + messaging + job queues,
      it's a critical single point of failure
    - Use Redis Sentinel or Redis Cluster in production
